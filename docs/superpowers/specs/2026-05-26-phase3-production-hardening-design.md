## Problem Statement

The Ledger API has a complete double-entry engine with hash-chain tamper detection and
financial reporting. It has no security layer, no observability, no deployment pipeline,
and no operational maturity signals. In 2026, AI writes feature code — interviewers now
probe engineering judgment in the disciplines AI can't generate without design intent:
security architecture, audit trails, deployment, observability. This phase adds all of
them.

## What This Phase Proves

- "How would you secure this in prod?" — API key auth, HMAC signing, rate limiting, threat model
- "How do you know when it's broken?" — structured logs, OTEL traces, Prometheus metrics, health endpoints
- "Can I see it running somewhere?" — live Fly.io deployment, GitHub Actions CI/CD
- "Prove nothing was missed" — immutable audit log with its own hash chain
- "What's your threat model?" — explicit document, OWASP API Top 10 coverage

## A. Security Foundation

### API Key Authentication

- Header: `X-API-Key: <key_id>.<secret>`
- Format: `<uuid>.<32-byte-random-hex>` — key_id is public (for lookup), secret is the credential
- Storage: `api_keys(id UUID, key_id TEXT, hashed_secret TEXT, scope TEXT, created_at, revoked_at, last_used_at)`
- `hashed_secret` = bcrypt(secret, cost=12). Why bcrypt: tunable brute-force cost, widely understood
- Every request: look up key_id, bcrypt.CompareHashAndPassword, check revoked_at is NULL
- On match: set `api_key_id` in request context for logging/audit

### Key Scoping

- `scope` field: `"read"` or `"write"` (extensible to `"admin"` later)
- Write scope required for: POST /v1/transactions, POST /v1/accounts, DELETE /v1/accounts
- Read scope sufficient for: all GET endpoints
- Enforced in middleware, not in handlers — scoping logic stays out of business code

### API Key Lifecycle

- `POST /v1/admin/api-keys` — create key, returns secret once (never stored plaintext)
- `DELETE /v1/admin/api-keys/{id}` — revoke (sets revoked_at, never hard-deletes)
- `POST /v1/admin/api-keys/{id}/rotate` — generates new secret, old secret valid 24h grace period, then revoked

### HMAC Request Signing (optional, recommended for demo)

- Client computes: `HMAC-SHA256(key=shared_secret, message=method+path+body_sha256+unix_timestamp)`
- Client sends: `X-Signature: <hmac_hex>`, `X-Timestamp: <unix_ts>`
- Server verifies: recompute HMAC, compare constant-time, reject if timestamp > 5 minutes old (replay protection)
- Demonstrates: crypto knowledge, replay attack awareness — common fintech interview topic

### Rate Limiting

- Sliding window counter per `api_key_id`, in-memory (`sync.Map` + `time.Ticker` cleanup)
- Default: 100 requests/minute per key
- Response on exceed: `429 Too Many Requests`, `Retry-After` header set, RFC 9457 error body (see Section E)
- Why in-memory: single-instance service, Redis adds complexity without benefit at this scale

### Security Headers Middleware

```
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Strict-Transport-Security: max-age=31536000; includeSubDomains
Content-Security-Policy: default-src 'none'
X-Request-ID: <correlation-id>
```

- Body size limit: 1MB via `http.MaxBytesReader`
- CORS: explicit allowlist (not `*`), configured via env var `CORS_ORIGINS`

### Container Hardening

- Dockerfile: `USER nonroot:nonroot` in runtime stage
- Final image: distroless/static (no shell, no package manager)
- `.gitignore` + CI check: fail if `.env` files are committed

## B. Observability

### Structured Logging (slog)

- Go stdlib `log/slog`, zero new dependencies
- Format: JSON in production (`slog.NewJSONHandler`), text in development
- Every request log line includes: `method`, `path`, `status`, `duration_ms`, `request_id`, `api_key_id`
- Error log lines include: `error`, `stack` (via `fmt.Errorf` wrapping)
- Log level configurable via `LOG_LEVEL` env var

### Correlation IDs

- Middleware generates UUID v4 if `X-Request-ID` not present in request
- Propagated through all log lines via context: `slog.With("request_id", id)`
- Returned in every response as `X-Request-ID` header
- DB queries include request_id in `SET LOCAL application_name` for Postgres log correlation

### Health Endpoints (unauthenticated)

```
GET /health    — liveness: returns 200 if process is alive
               {"status":"ok","version":"1.2.3","uptime_s":3600}

GET /readiness — readiness: DB ping + pool stats
               {"status":"ok","db_ping_ms":2,"pool_idle":4,"pool_total":10}
```

- Used by Fly.io health checks and any load balancer
- Never requires auth — external probes cannot authenticate

### Prometheus Metrics

- Dependency: `github.com/prometheus/client_golang` (small, stable, industry standard)
- Metrics exposed at `GET /metrics` (auth-gated, read-scope only)
- Counters: `ledger_requests_total{method,path,status}`, `ledger_api_key_auth_failures_total`
- Histograms: `ledger_request_duration_seconds{method,path}`, `ledger_db_query_duration_seconds{query}`
- Gauges: `ledger_db_pool_idle`, `ledger_db_pool_total`
- No Grafana setup required — metrics endpoint alone demonstrates Prometheus knowledge

### OpenTelemetry Tracing

- Dependency: `go.opentelemetry.io/otel` (stable in 2026, the industry standard)
- Every request: root span from HTTP handler, child spans for DB queries
- Span attributes: `http.method`, `http.route`, `db.statement` (parameterized, no values), `api_key.id`
- Exporter: stdout OTLP JSON in dev (`OTEL_EXPORTER=stdout`), configurable for prod
- Why OTEL: replaces vendor-specific tracing (Datadog, Jaeger) with portable instrumentation

## C. Forensics / Audit Log

### Schema

```sql
CREATE TABLE audit_events (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    event_type  TEXT NOT NULL,           -- 'account.created', 'transaction.posted', etc.
    actor       TEXT NOT NULL,           -- api_key_id or 'system'
    resource_type TEXT NOT NULL,         -- 'account', 'transaction', 'api_key'
    resource_id TEXT NOT NULL,           -- UUID of the affected resource
    payload_json JSONB NOT NULL,         -- snapshot of what changed (before/after for mutations)
    prev_hash   TEXT NOT NULL,           -- SHA-256 of previous audit event (chain-linked)
    hash        TEXT NOT NULL,           -- SHA-256 of this event's canonical fields
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### Append-Only Enforcement

- Postgres Row Level Security: `DENY UPDATE, DELETE ON audit_events TO ledger_app_role`
- Application never calls UPDATE or DELETE on this table
- Separate DB role `ledger_app_role` with INSERT + SELECT only on audit_events

### Chain Linking

- Same pattern as transaction hash chain: each event includes `prev_hash` of previous event
- First event: `prev_hash = "genesis"`
- On insert: read latest hash, compute `SHA-256(id + event_type + actor + resource_id + payload_json + prev_hash + created_at)`, store as `hash`
- `GET /v1/admin/audit-events/verify` — re-walks chain, verifies every hash links correctly

### Covered Events

| Event type | Trigger | Payload |
|---|---|---|
| `account.created` | POST /v1/accounts | account snapshot |
| `account.archived` | DELETE /v1/accounts/{id} | account snapshot |
| `transaction.posted` | POST /v1/transactions | transaction + entries snapshot |
| `api_key.created` | POST /v1/admin/api-keys | key_id + scope (no secret) |
| `api_key.revoked` | DELETE /v1/admin/api-keys/{id} | key_id |
| `api_key.rotated` | POST /v1/admin/api-keys/{id}/rotate | key_id |
| `chain.verified` | GET /v1/chain/verify | result + chain_head |

## E. System Design Improvements

### Request Timeout Middleware

- Every request gets `context.WithTimeout(ctx, 10s)`
- Handler must respect context cancellation — `ctx.Done()` propagated to DB queries
- On timeout: `503 Service Unavailable` with `Retry-After: 1`
- Prevents slow DB queries from holding connections indefinitely

### Circuit Breaker (DB)

- Dependency: `github.com/sony/gobreaker` (tiny, well-maintained)
- Wraps all DB calls: after 5 consecutive errors within 10s, circuit opens
- Open state: immediate `503` without hitting DB — protects DB from thundering herd during outages
- Half-open after 30s: allows one probe request to test recovery

### DB Connection Pool Tuning

```go
pgxpool.Config{
    MaxConns:          int32(max(4, runtime.NumCPU())),
    MinConns:          2,
    MaxConnLifetime:   30 * time.Minute,
    MaxConnIdleTime:   5 * time.Minute,
    HealthCheckPeriod: 1 * time.Minute,
}
```

- Logged at startup: `slog.Info("db pool configured", "max_conns", n, "min_conns", 2)`
- Pool stats exposed via `/readiness` and Prometheus metrics

### API Versioning

- All routes prefixed `/v1/`: `r.Route("/v1", func(r chi.Router) { ... })`
- No breaking changes yet, but the pattern is set for future `/v2/` alongside `/v1/`
- `Deprecation` header added to any endpoint scheduled for removal

### Migrations Consolidation

- Currently two identical `migrations/` directories (root and `internal/store/migrations/`)
- Consolidate to `internal/store/migrations/` as single source of truth
- Root `migrations/` removed; `migrate.go` updated to embed from single path

### Soft Delete for Accounts

- Add `archived_at TIMESTAMPTZ` column to `accounts` table (migration 005)
- `DELETE /v1/accounts/{id}` sets `archived_at = NOW()`, never hard-deletes
- All queries filter `WHERE archived_at IS NULL` by default
- Why: financial data is never truly deleted; auditors expect it

### Idempotency Key TTL

- Add background goroutine: every 1h, `DELETE FROM idempotency_keys WHERE created_at < NOW() - INTERVAL '7 days'`
- Started in `serve.go`, stopped on graceful shutdown via context cancellation
- Prevents unbounded table growth

### Graceful Shutdown

```go
quit := make(chan os.Signal, 1)
signal.Notify(quit, syscall.SIGTERM, syscall.SIGINT)
<-quit
ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
defer cancel()
server.Shutdown(ctx)   // drains in-flight requests
pool.Close()           // closes DB connections
```

### RFC 9457 Problem Details Error Format

Replace `{"error": "invalid date"}` with:

```json
{
  "type": "https://ledger.example.com/errors/invalid-date",
  "title": "Invalid Date Format",
  "status": 400,
  "detail": "Parameter 'from' must be YYYY-MM-DD, got '2026/05/01'",
  "instance": "/v1/reports/pl?from=2026/05/01"
}
```

- Standard per IETF RFC 9457. Banks increasingly require this. Interviewers notice it.
- Centralized `problem.Write(w, status, type, title, detail)` helper function

## F. Fintech-Specific Patterns

### Threat Model Document

Written to `docs/threat-model.md`. Covers:

- **Assets**: transaction data, account data, audit log, API keys
- **Threats** (STRIDE):
  - Spoofing: mitigated by API key auth + bcrypt
  - Tampering: mitigated by hash chain on transactions + audit log
  - Repudiation: mitigated by immutable audit log
  - Information disclosure: mitigated by key scoping, no PII in logs
  - DoS: mitigated by rate limiting, request timeout, circuit breaker
  - Elevation of privilege: mitigated by key scoping (read vs write)
- **Not mitigated** (explicitly documented): multi-tenancy, mTLS, HSM key storage

### Secrets Management

- Dev: `.env` file (gitignored), loaded via `os.Getenv`
- Prod: Fly.io secrets (encrypted at rest, injected as env vars at runtime)
- Documented Vault path for enterprise context: `secret/ledger/{env}/db_url`
- Pre-commit hook: `grep -r "postgres://" --include="*.go"` fails if hardcoded DSN found

### Column-Level Encryption

- `pgcrypto` extension: `CREATE EXTENSION IF NOT EXISTS pgcrypto`
- `account.description` encrypted at rest: `pgp_sym_encrypt(description, $ENCRYPTION_KEY)`
- Decrypt on read: `pgp_sym_decrypt(description, $ENCRYPTION_KEY)::text`
- `ENCRYPTION_KEY` sourced from env var, never stored in DB
- Why not full TDE: Postgres TDE is enterprise-only; column encryption is practical and demonstrable

### SBOM (Software Bill of Materials)

- `syft . --output spdx-json > sbom.spdx.json` in CI
- Published as GitHub Actions artifact on every build
- Why: enterprise clients (banks) increasingly require SBOM for vendor security reviews

## NOT in Scope

- Multi-tenancy / organization isolation
- JWT / OAuth2 (no users/sessions in this system — API keys are correct here)
- mTLS service-to-service (single service, no internal mesh)
- HSM key storage (enterprise feature, out of scope for portfolio)
- Redis for distributed rate limiting (single instance, in-memory is correct here)
- Grafana dashboard (metrics endpoint is sufficient signal)
- Load testing / chaos engineering
- Multi-region deployment

## Success Criteria

- `curl https://ledger-api.fly.dev/health` returns `{"status":"ok"}` from a public URL
- API without `X-API-Key` header returns `401`; wrong key returns `401`; 101st req/min returns `429`
- Corrupt a transaction row → `GET /v1/chain/verify` returns tampered, `GET /v1/admin/audit-events/verify` still intact (different chains)
- `govulncheck ./...` passes in CI with no known CVEs
- All request logs include `request_id`, `api_key_id`, `duration_ms`
- OTEL span visible in stdout when `OTEL_EXPORTER=stdout`
- Threat model doc exists at `docs/threat-model.md`
- SBOM artifact published on every CI run
